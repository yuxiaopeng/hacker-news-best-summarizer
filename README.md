## 2. 2025年：LLM之年

**原文标题**: 2025: The Year in LLMs

**原文链接**: [https://simonwillison.net/2025/Dec/31/the-year-in-llms/](https://simonwillison.net/2025/Dec/31/the-year-in-llms/)

2025年是大型语言模型（LLM）的里程碑式一年，多个方面都取得了显著进展。“推理之年”见证了OpenAI的o1-o4 mini和DeepSeek R1等模型利用可验证奖励强化学习（RLVR）开发出复杂的解决问题策略，在工具使用、AI辅助搜索和代码调试方面表现出色。

这种智能推动了“智能体之年”的到来，尤其是以Anthropic的Claude Code为代表的“编码智能体”。这些系统能够编写、执行和迭代代码，既可以作为命令行工具使用，也可以作为异步网络平台（如Claude Code for web和OpenAI的Codex Cloud）提供。这些命令行智能体（Claude Code营收达10亿美元）产生的可观收入突显了它们的实用性，不过以“YOLO模式”（自动确认）运行它们也凸显了与“异常常态化”相关的安全问题。

在财务方面，每月200美元的订阅服务（例如Claude Pro Max 20x、ChatGPT Pro）成为新趋势，以满足重度用户的需求。一个主要进展是DeepSeek、Qwen、Kimi和GLM等“顶尖中国开源模型”的崛起。这些模型通常以开源许可发布，成为了世界领先者，甚至影响了全球市场，并推动了高效训练技术的发展。

大型语言模型在处理“长任务”方面也取得了显著进展，GPT-5和Claude Opus 4.5等模型能够完成复杂、耗时数小时的软件工程挑战。这一年还见证了“提示词驱动的图像编辑”消费级产品的成功推出。

## 3. 毁灭所有网站的网站

**原文标题**: A website to destroy all websites

**原文链接**: [https://henry.codes/writing/a-website-to-destroy-all-websites/](https://henry.codes/writing/a-website-to-destroy-all-websites/)

文章《一个可以摧毁所有网站的网站》指出，互联网已经从一个“广阔、神圣的自我发现与愉悦之地”退化成一个负面空间，充斥着“注意力农场”式内容、社交媒体陷阱以及工业规模的网页开发。作者将过去那个专家教学、真挚友谊和创意编程蓬勃发展的互联网，与现在这个充斥着AI生成内容、广告泛滥和缺乏灵感的网页工程的互联网进行了对比。

该文借鉴伊万·伊里奇的《共生工具论》，将这种衰落描述为一种普遍模式：技术最初提升人类自由，但随后工业化，形成“激进垄断”，导致社会反过来服务于工具。汽车、纺织业、抗生素和太空探索都是例证。同样，万维网也演变成一个旨在榨取资本和注意力的系统，从而引发虚假信息和成瘾等问题。

根据伊里奇的框架，解决方案在于“共生工具”——即可持续的、增强自主性的和本地优先的技术。应用于互联网，这意味着：
*   **教与学：** 摒弃YouTube或Substack等大型一体化平台，回归个人博客（HTML、RSS），以重获内容所有权并促进真正的思想交流。
*   **与朋友联系：** 使用去中心化协议（POSSE、ActivityPub）构建个人社交网络，以创建持久的、由所有者控制的、不受公司左右的社区。
*   **为万维网编程：** 通过HTML、CSS和渐进增强的JavaScript，将网页工程重新塑造成一项创意工艺，鼓励实验而非工业化生产。

最终，文章总结道，重塑互联网的关键在于**个人网站**。这些手工编码、可联合发布、高度个性化的网站，体现了自主性、所有权和创意表达，为工业化、“糟糕”的互联网提供了一个有力的替代方案。

## 5. 为什么用户无法直接创建问题

**原文标题**: Why users cannot create Issues directly

**原文链接**: [https://github.com/ghostty-org/ghostty/issues/3558](https://github.com/ghostty-org/ghostty/issues/3558)

Ghostty 项目要求用户在提交 Issue（问题）前先创建 Discussion（讨论）。这项政策源于多年的开源维护经验，旨在优化问题追踪器，使其专注于可执行的工作项。

与许多项目不同，Ghostty 使用 GitHub Discussions 来进行一般讨论、功能请求和初步问题报告。只有当讨论演变为一个清晰、可操作的项目时，维护者才会将其转换为 Issue。这种方法确保了问题追踪器中的每一个事项都已准备好进行开发，从而提高了贡献者的效率。

维护者指出，用户报告的“bug”（错误）中，有很高比例（80-90%）通常是误解、环境问题或配置错误。其余大部分通常是规格不明确的功能请求。通过先在讨论中进行筛选，团队可以提供指导并完善提交内容。

识别出有效且可重现问题的用户，其讨论将由维护者转换为 Issue，确保他们无需为已确认的问题付出额外工作。更多详情，用户可查阅 `CONTRIBUTING.md` 文件。

奥斯汀·Z·亨利，卡内基梅隆大学的副教学教授，讲述了他于2023年初与一家大型出版商签约后又取消图书合约的经历。最初，他被撰写一本关于经典编程项目的机会所吸引，这是他长期以来的一个目标。他权衡了传统出版的利弊，尽管出版商提供了结构和发行渠道，但亨利觉得财务条款（低预付款、版税）没有吸引力，并指出他们的营销投入极少。

由于理念不合，合作很快就恶化了。出版商反复要求他“降低内容难度”，压抑他的个人风格，并加入一章Python入门内容，这反映了一种程式化的做法。一个主要的摩擦点是出版商坚持在每本书中都整合人工智能，亨利对此坚决拒绝，因为它与他书中永恒、基础项目的核心理念相悖。

最终，亨利不堪重负且不再享受这一过程，他要求暂停项目。出版商最终终止了合约，并将所有权利归还给他。尽管面临挑战，亨利仍然对他写书的想法充满热情，并决定自行出版，提供电子书预售，各章节完成后即发布。

## 7. 我是一名头部外卖应用的开发者。

**原文标题**: I'm a developer for a major food delivery app

**原文链接**: [https://old.reddit.com/r/confession/comments/1q1mzej/im_a_developer_for_a_major_food_delivery_app_the/](https://old.reddit.com/r/confession/comments/1q1mzej/im_a_developer_for_a_major_food_delivery_app_the/)

Reddit上的一篇帖子，标题为《我是一家大型外卖应用的开发者。我们对待员工（送货司机）的方式糟透了，我却对此无能为力》，从内部人士的角度详细阐述了对外卖配送员的严重剥削。这位开发者对公司的做法深感愧疚和无力。

帖子中强调的几个主要问题包括：

*   **低薪：** 司机薪酬极低，一旦计入燃油和车辆维护等开销，收入常常低于最低工资标准。长距离配送的报酬尤其不合理。
*   **批量派单：** 应用程序经常将多个订单合并给一名司机，但薪酬增幅微乎其微，迫使司机做更多工作，却只获得不成比例的微薄额外报酬，尽管顾客支付了单独的配送费。
*   **不公平的评分系统：** 如果客户评分过低，司机可能会被停用，即使评分受到司机无法控制的因素影响，例如餐厅延迟出餐或食物质量问题。
*   **缺乏支持：** 当出现问题（例如地址错误、延迟取消）时，司机首当其冲，常常损失时间、燃油和报酬，且无任何补偿。客户支持部门也常常偏袒客户而非司机。
*   **独立承包商身份：** 司机被归类为独立承包商，这使他们无法享受福利、最低工资保障和工作稳定性。

这位开发者在开发加剧这种剥削系统的功能时感到自己是帮凶，但由于自身就业需求，无力改变底层的商业模式。
