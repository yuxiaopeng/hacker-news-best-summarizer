## 7. 脏碎片：通用 Linux 本地权限提升

**原文标题**: Dirty Frag: Universal Linux LPE

**原文链接**: [https://www.openwall.com/lists/oss-security/2026/05/07/8](https://www.openwall.com/lists/oss-security/2026/05/07/8)

本报告详细介绍了“Dirty Frag”，这是一个严重的通用本地权限提升（LPE）漏洞，影响所有主流Linux发行版，允许非特权用户获取root权限。该漏洞由Hyunwoo Kim发现，其影响与此前的“Copy Fail”漏洞相当。

由于禁令被违反，Dirty Frag的公开披露过早发生，这意味着目前没有官方补丁或CVE可用。该漏洞串联了两个独立的与网络功能相关的内核问题，并提供了具体的内核提交链接。

一种临时缓解措施是，通过向`/etc/modprobe.d/dirtyfrag.conf`添加规则来禁用`esp4`、`esp6`和`rxrpc`内核模块，然后将其移除。

所提供的概念验证（PoC）漏洞利用代码演示了LPE。它利用了非特权用户和网络命名空间（`CLONE_NEWUSER | CLONE_NEWNET`），以及`xfrm` (IPSec) 和`rxrpc`内核模块中的漏洞。该漏洞利用构建了一个旨在执行root shell的小型ELF载荷，然后使用网络套接字操作（涉及`xfrm_usersa_info`操作和通过UDP封装进行的`splice`调用）将此载荷直接写入`/usr/bin/su`二进制文件的页缓存中。这种覆写替换了目标的初始指令，从而使得`/usr/bin/su`的后续执行能够立即授予root权限。

## 8. 谷歌云防欺诈不过是重新包装的WEI。

**原文标题**: Google Cloud Fraud Defence is just WEI repackaged

**原文链接**: [https://privatecaptcha.com/blog/google-cloud-fraud-defence-wei/](https://privatecaptcha.com/blog/google-cloud-fraud-defence-wei/)

Google Cloud 欺诈防御于2026年5月推出，被宣传为“reCAPTCHA的下一代演进”，实则是对2023年曾被否决的“Web环境完整性”（WEI）提案的重新包装。WEI旨在让设备硬件以加密方式证明浏览器和设备的完整性，但在Mozilla和电子前哨基金会的强烈反对下被撤回，两方均称其为“Chrome的Web数字版权管理（DRM）计划”，并认为它对开放互联网和用户隐私构成威胁。

欺诈防御采用二维码挑战，要求用户使用“安装了Google Play服务的新型安卓设备，或新型iPhone/iPad”进行扫描。文章强调，“安装Google Play服务”是关键，因为它利用Play Integrity API进行设备证明——这与WEI的核心机制相同。与WEI不同的是，欺诈防御在未经公众审查的情况下商业发布。

主要批评包括：
1.  **可规避性：** 机器人可以使用廉价的认证安卓设备或自动化摄像头装置，轻易地规避二维码挑战。
2.  **网络钓鱼风险：** 它培养用户扫描二维码的习惯，使他们容易受到恶意网络钓鱼活动的攻击。
3.  **排他性：** 运行定制安卓ROM（例如GrapheneOS、未安装Play服务的LineageOS）的注重隐私的用户，或使用Firefox for Android（未集成Play Integrity）等浏览器的用户将被有效阻止。
4.  **追踪：** 每一次成功的挑战都会向Google发送信号，告知哪个认证设备访问了哪个网站，从而创建一个持久性标识符，并累积用户的全面浏览活动记录，引发重大的隐私担忧。

文章总结道，欺诈防御是一个设备证明系统，而不仅仅是reCAPTCHA的更新。它规避了此前的公众审查流程，旨在建立一个曾被否决的归因系统，同时未能提供针对机器人的强大解决方案。

## 9. A web page that shows you everything the browser told it without asking

**原文标题**: A web page that shows you everything the browser told it without asking

**原文链接**: [https://sinceyouarrived.world/taken](https://sinceyouarrived.world/taken)

The web page "Since You Arrived · Vol. IV" aims to transparently demonstrate the extensive personal data your browser automatically discloses to any website, "without asking." It emphasizes that while this information is constantly disclosed, JavaScript is essential for the page to *show* you what it has learned.

The page utilizes standard browser APIs and documented techniques, not exploits, to gather data. This includes your IP address (used for approximate geolocation via a third-party, though only partially displayed), detailed device specifications (screen, GPU, battery status, language, operating system, installed fonts), and browser characteristics. It highlights font fingerprinting as a strong signal for unique user identification and references research on how battery data can be used for tracking.

A unique "barcode" is generated based on this device data, computed locally in your browser and never transmitted. Crucially, the page clarifies several prevalent tracking methods it *did not* employ, such as canvas fingerprinting, accessing your clipboard, or detecting logged-in accounts on other sites, while noting these are commonly used elsewhere.

The creators stress their commitment to privacy: they do not track, store any data (no cookies, local storage), or log personal information. Only two anonymous events (arrival, finish) are sent to their server, with no identifying data retained. The content is hand-written, not AI-generated. The overarching message is to expose how easily users can be uniquely identified across the web due to inherent browser design, urging awareness of what every page silently learns about you.
