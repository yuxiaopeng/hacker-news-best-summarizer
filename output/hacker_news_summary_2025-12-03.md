## 1. 安索罗匹克收购邦

**原文标题**: Anthropic acquires Bun

**原文链接**: [https://bun.com/blog/bun-joins-anthropic](https://bun.com/blog/bun-joins-anthropic)

Anthropic 已收购 Bun，这个高性能一体化的 JavaScript 运行时、打包器和包管理器。Anthropic 将把 Bun 集成为其 AI 编程产品（包括 Claude Code 和 Claude Agent SDK）的基础设施，相信它将为未来的 AI 开发工具提供动力。

至关重要的是，Bun 将保持开源、MIT 许可，并继续由原团队在 GitHub 上公开开发。其发展路线图仍将优先考虑高性能 JavaScript 工具、Node.js 兼容性，以及取代 Node.js 作为服务器端运行时。

此次收购带来了多重益处：Bun 将有助于使 AI 编程工具更快、更精简，获得对 AI 编程未来发展的早期洞察，并加速自身发展。对于 Bun 而言，这意味着获得 Anthropic 提供的长期稳定性和资源，使其无需独立将产品商业化。反过来，Anthropic 获得了一个强大的运行时，与 AI 驱动的软件开发不断演进的格局相契合，在这一领域，快速、可预测的运行时对于 AI 智能体编写和部署代码至关重要。

Bun 的发展始于五年前，旨在解决缓慢的开发周期，并发展成为一个迅速广受欢迎的综合性 JavaScript 工具包。其单文件可执行文件已被证明是分发 AI 命令行界面（CLI）工具的理想选择，因此被 Claude Code 等产品采用。此次战略收购确保了 Bun 的持续增长，同时将其定位在 AI 驱动的软件开发前沿。

## 2. Zig 退出 GitHub，称微软的 AI 痴迷毁了该服务

**原文标题**: Zig quits GitHub, says Microsoft's AI obsession has ruined the service

**原文链接**: [https://www.theregister.com/2025/12/02/zig_quits_github_microsoft_ai_obsession/](https://www.theregister.com/2025/12/02/zig_quits_github_microsoft_ai_obsession/)

Zig软件基金会已正式退出GitHub，将其项目迁移至Codeberg，理由是工程卓越性下降和微软对“AI的痴迷”。Zig总裁Andrew Kelly表示，GitHub Actions存在“不可原谅的bug”和“凭感觉的调度”，导致CI系统严重积压。

一个主要例子是2022年2月引入的“safe_sleep.sh”脚本错误，该错误导致CI运行器无限期挂起并占用100%的CPU。这一严重缺陷于2025年4月被公开讨论，但修复方案直到2025年8月才从另一个问题中合并，而最初的bug报告则一直开放到2025年12月。Answer.AI联合创始人Jeremy Howard将这种情况描述为“一系列令人咋舌的低级失误”，表明存在根本性的组织问题。

继Zig离开之后，Dillo浏览器开发者Rodrigo Arias Mallo也宣布计划离开GitHub，原因是担心对JavaScript的过度依赖、潜在的服务中断、可用性下降、审核不力以及GitHub对生成式AI的关注正在“摧毁开放网络”。

此次出走恰逢替代平台Codeberg的用户数量翻倍。尽管GitHub对这些批评保持沉默，微软CEO萨蒂亚·纳德拉仍在强调GitHub Copilot用户的快速增长及其对GitHub收入的巨大贡献，这突显了向AI整合的战略转变，尽管核心开发服务据称正在退化。

## 3. 采用美国汽车标准将危及欧洲人生命

**原文标题**: Accepting US car standards would risk European lives

**原文链接**: [https://etsc.eu/accepting-us-car-standards-would-risk-european-lives-warn-cities-and-civil-society/](https://etsc.eu/accepting-us-car-standards-would-risk-european-lives-warn-cities-and-civil-society/)

巴黎、布鲁塞尔和阿姆斯特丹等多个城市，以及超过75个民间社会组织组成的联盟，正在敦促欧盟立法者放弃其在贸易协议中接受美国较低汽车标准的意图。他们警告说，与美国更宽松的规定对齐将损害欧洲在道路安全、公共健康、气候政策和竞争力方面的全球领导地位。

签署方强调了道路安全方面的鲜明对比：自2010年以来，由于强制性生命安全技术，如行人保护和自动紧急制动，欧盟的道路死亡人数下降了36%。同期，美国的道路死亡人数上升了30%，其中行人和骑自行车者的死亡人数分别增加了80%和50%。欧盟基本的行人保护规则（使特斯拉Cybertruck等车辆非法）将受到威胁，从而毁掉数十年的进步。

公共健康也面临风险。欧盟计划从2026年起对制动器和轮胎磨损产生的有害污染物施加限制，而美国正在放宽车辆的空气污染规则。采纳美国标准将增加欧洲人接触与严重呼吸道和心血管疾病相关的污染物。

从经济角度看，接受美国较低的标准可能会威胁欧洲的就业。主要的欧盟品牌已经在其美国工厂生产符合欧盟标准的车辆。如果欧洲接受美国较低的标准，生产可能会从欧盟转移到美国，导致欧洲汽车工厂及其供应链的大规模失业。这些组织还指出，目前对美国超大型卡车的进口漏洞（已在被滥用）将显著扩大。

签署方敦促欧盟立法者反对这一意图，并公开声明欧盟的汽车标准不容谈判。

## 7. 国会议员选股表现高出47个百分点。

**原文标题**: Congressional lawmakers 47% pts better at picking stocks

**原文链接**: [https://www.nber.org/papers/w34524](https://www.nber.org/papers/w34524)

NBER最新工作论文《国会山上的“船长收益”》（由魏尚进和周轶帆撰写）揭示，担任领导职务的美国国会议员表现出卓越的选股能力。该研究分析了交易层面的数据，发现这些议员在升任领导职务之前，表现与其同僚相似，但在担任领导职务 *之后*，他们每年平均跑赢同僚47个百分点。

## 10. 苹果发布开源权重视频模型

**原文标题**: Apple Releases Open Weights Video Model

**原文链接**: [https://starflow-v.github.io](https://starflow-v.github.io)

苹果公司发布了STARFlow-V，这是一款基于归一化流（NFs）的开创性视频生成模型，旨在视觉质量上与扩散模型相媲美。这标志着归一化流首次在因果视频生成中展现出这种能力，并提供了端到端训练、精确似然估计以及对文本到视频（T2V）、图像到视频（I2V）和视频到视频（V2V）任务的原生多任务支持等优点。

STARFlow-V采用了一种全局-局部架构，其中一个深度因果Transformer在压缩的潜在空间中处理全局时间推理，而浅层流块则管理局部帧内细节。这种设计最大限度地减少了误差累积。主要创新包括流-分数匹配（Flow-Score Matching），该技术整合了一个轻量级因果去噪器以提高视频一致性；以及视频感知雅可比迭代（Video-Aware Jacobi Iteration），通过并行更新来提高采样效率。

该70亿参数模型使用7000万文本-视频对和4亿文本-图像对进行训练，能够以16fps生成480p视频。其固有的可逆结构使其无需改变架构即可无缝支持各种生成任务。经验证明，STARFlow-V实现了强大的视觉保真度和时间一致性，这使得归一化流成为高质量自回归视频生成和“世界模型”开发的一个极具前景的方向。

## 12. 十亿行数据，十万 TPS：SQLite 的不合理有效性

**原文标题**: 100k TPS over a billion rows: the unreasonable effectiveness of SQLite

**原文链接**: [https://andersmurphy.com/2025/12/02/100000-tps-over-a-billion-rows-the-unreasonable-effectiveness-of-sqlite.html](https://andersmurphy.com/2025/12/02/100000-tps-over-a-billion-rows-the-unreasonable-effectiveness-of-sqlite.html)

本文推崇 SQLite 的“不合理有效性”，认为其嵌入式、单写入器的特性，通常被视为一种局限，但实际上却能为单服务器上的交互式事务提供卓越的性能。

作者在 M1 Pro MacBook Pro 上使用 Clojure 基准测试，在十亿行数据集和幂律用户分布下，将 SQLite 与 Postgres 等网络数据库进行对比。Postgres 在本地环境下达到 13756 TPS，但模拟 10 毫秒网络延迟后，性能骤降至 702 TPS。应用 `SERIALIZABLE` 隔离级别并增加查询量后，性能进一步降至 348 TPS，这表明阿姆达尔定律和网络开销如何为网络数据库造成了硬性瓶颈。

SQLite 通过消除网络延迟，立即达到 44096 TPS。其单写入器设计使得简单的动态批处理成为可能，将性能提升至 186157 TPS。即使使用 `SAVEPOINT`（嵌套事务）进行细粒度事务回滚，SQLite 仍保持 121922 TPS。在模拟 75% 并发读取的混合工作负载时，它依然实现了令人印象深刻的 102545 TPS。

后续更新解决了对 `synchronous "NORMAL"` 模式的担忧，表明即使使用 `synchronous "FULL"` 以获得更强的持久性，批处理和并发读取下的 SQLite 仍能提供超过 100k TPS 的性能。尝试优化 Postgres（例如，更大的连接池）通常会得到喜忧参半甚至更差的可串行化结果。

## 17. Zig的异步程序新方案

**原文标题**: Zig's new plan for asynchronous programs

**原文链接**: [https://lwn.net/SubscriberLink/1046084/4c048ee008e1c70e/](https://lwn.net/SubscriberLink/1046084/4c048ee008e1c70e/)

Zig 为异步程序引入了一项新方案，旨在解决 Python 和 Rust 等语言中普遍存在的“函数着色问题”（即异步代码需要特殊语法）。Zig 的极简主义方法现在将异步操作隐藏在一个新的通用 `Io` 接口之后。

执行 I/O 的函数接受一个 `Io` 实例作为参数。标准库提供了 `Io.Threaded` 用于同步执行（通过线程实现显式并行），以及 `Io.Evented` 用于通过事件循环（例如 `io_uring`）实现异步 I/O。关键在于，*同一个函数* 可以根据传入的 `Io` 实例同步或异步运行，而无需改变其函数签名或实现。

为了可选的性能提升，如果 `Io` 实现支持，`io.async()` 允许操作并行运行。对于必需的并发执行（例如，在处理用户输入的同时监听端口），`io.concurrent()` 明确请求并行处理。

这种设计使异步性成为透明的性能优化，而非根本性的 API 变更。尽管传递 `Io` 参数会增加一些冗余，但这符合 Zig 的惯用法，保留了典型的控制流（如 `try` 和 `defer`），并增强了可测试性和控制力。`Io.Evented` 的实现仍处于实验阶段，但总体设计为 Zig 走向 1.0 版本提供了一种灵活且面向未来的方法。

## 25. 应对新增情况

**原文标题**: Addressing the adding situation

**原文链接**: [https://xania.org/202512/02-adding-integers](https://xania.org/202512/02-adding-integers)

本文解释了x86编译器如何高效地执行整数加法，尽管该架构的`add`指令存在双操作数限制（例如`lhs += rhs`），这会覆盖源寄存器，并且不像ARM那样允许指定一个独立的目的地寄存器。

编译器通过“滥用”`lea`（加载有效地址）指令来克服这一限制。`lea`指令旨在利用复杂的寻址模式（例如`[寄存器 + 寄存器 * 比例因子 + 偏移量]`）计算内存地址，而无需实际访问内存。通过将这种地址计算机制视为一个通用算术单元，`lea`可以用于执行三操作数加法。例如，`lea eax, [rdi + rsi]`有效地将`rdi`和`rsi`相加，并将结果存储到`eax`中，同时保持`rdi`和`rsi`的值不变。

这种基于`lea`的优化通常能节省一条指令，保留原始操作数的值，并允许在x86处理器上实现更高效的执行。尽管`lea`计算的是一个64位地址，但在满足ABI要求的前提下，当结果写入较小的目的寄存器（如32位`eax`）时，其高位部分会被丢弃。编译器会自动实现这些复杂的优化，使开发人员无需手动关注。

## 42. 逆数学揭示难题为何难

**原文标题**: Reverse math shows why hard problems are hard

**原文链接**: [https://www.quantamagazine.org/reverse-mathematics-illuminates-why-hard-problems-are-hard-20251201/](https://www.quantamagazine.org/reverse-mathematics-illuminates-why-hard-problems-are-hard-20251201/)

“逆向数学”是一种元数学技术，它阐明了为什么一些困难的计算问题至今仍未解决。计算机科学家长期以来一直难以从数学上证明像旅行商问题这类问题的固有难度。如今，他们正在利用元数学分析证明过程本身，旨在理解当前方法的局限性。

在逆向数学中，研究人员颠覆了常规方法：他们不再是从公理推导定理，而是用一个定理替换一个公理，然后证明这个公理，通常在一个较弱的公理系统（如PV1）中进行。这有助于揭示不同数学陈述之间精确的逻辑关系。

陈立杰、李家图和伊戈尔·奥利维拉将此方法应用于复杂性理论，发现许多表面上不同的定理在PV1中是逻辑等价的。例如，他们发现基础的鸽巢原理等价于“相等性问题”（判断两个二进制字符串是否相同）的下限。最令人惊讶的是，他们表明它也等价于单带图灵机识别回文串的下限——这是简单的计数原理与特定计算问题之间的联系。

这些等价性表明，许多复杂性下限比以前认为的更为基础且相互关联更深。尽管这项工作尚未直接解决困难问题，但它突出了PV1等公理系统的局限性，并重新激发了人们对元数学的兴趣，将其视为理解计算复杂性基础挑战的一种方式。

## 43. 我为何停止在我的API中使用JSON

**原文标题**: Why I stopped using JSON for my APIs

**原文链接**: [https://aloisdeniel.com/blog/better-than-json](https://aloisdeniel.com/blog/better-than-json)

文章解释了尽管JSON无处不在，作者为何在API开发中选择Protocol Buffers（Protobuf）而非JSON。作者承认JSON的优势——可读性强、与Web深度集成、灵活性高和丰富的工具——但同时强调了它的缺点：缺乏严格性，这可能导致字段缺失、类型不正确或结构未文档化。

由谷歌开发的Protobuf提供了一个强大的替代方案。它的主要优势是强类型，通过`.proto`文件强制执行，这些文件精确定义了具有严格类型和唯一数字标识符的消息架构。这使得代码能够跨多种语言（如Dart、TypeScript、Go等）自动生成，确保端到端的类型安全，省去手动验证，并显著减少与数据相关的错误。

此外，Protobuf是一种超高效的二进制序列化格式。由于优化编码、使用数字标签而非文本键以及没有空格等开销，它通常比JSON紧凑3倍。这种紧凑性带来了更低的带宽消耗、更快的响应时间以及改善的用户体验。作者展示了Protobuf如何独立地与传统HTTP API配合使用，并提供了一个Dart示例，以实现无缝、类型安全的通信。

Protobuf的主要缺点是其二进制特性，与JSON不同，如果没有专用工具和相应的架构，人类无法直接阅读。然而，作者认为，与性能、健壮性以及更愉快的开发体验方面获得的巨大收益相比，这种权衡是值得的，并强烈推荐Protobuf用于追求效率和减少错误的项目。

1.  **套接字创建：** 使用 `socket2` crate 初始化一个 `IPV4` 数据报套接字 (`Type::DGRAM`)，并明确设置 `Protocol::ICMPV4`。随后，此 `socket2::Socket` 被转换为一个 `std::net::UdpSocket`。
2.  **数据包创建与发送：** 构造一个 ICMP 回显请求数据包。一个关键区别在于平台特定的行为：Linux 内核会覆盖标识符和校验和字段，而 macOS 则要求它们被正确计算并提供。随后，该数据包被发送到目标 IP。
3.  **响应接收与解析：** 接收响应的方式也因操作系统而异。macOS 在接收到的数据中包含 IP 头（20 字节），在解析 ICMP 有效载荷之前必须将其剥离。Linux 则不包含此头。在对头部进行调整（如果存在）后，可以提取并验证 ICMP 回复类型、序列号和有效载荷。

**原文标题**: Why are my headphones buzzing whenever I run my game?

**原文链接**: [https://alexene.dev/2025/12/03/Why-do-my-headphones-buzz-when-i-run-my-game.html](https://alexene.dev/2025/12/03/Why-do-my-headphones-buzz-when-i-run-my-game.html)

作者发现，尽管拥有一台强大的PC和USB供电的DAC，但其耳机在运行自制等距游戏时会持续发出嗡嗡声。其他图形密集型游戏并未引起此类问题。作者最初怀疑是电源问题，但令其困惑的是，相比其他高要求大作，反而是自己这个简单的游戏更容易触发此问题。

调试显示，当禁用GPU渲染时，嗡嗡声便会停止。该游戏的图形管线包含一个“拾取纹理”，用于存储鼠标选择的实体ID。关键在于，为了检查点击，这个完整的纹理每帧都会从GPU复制到CPU内存。

通过系统地禁用管线组件，问题被定位到此拾取纹理的GPU到CPU传输。改变传输频率会直接影响嗡嗡声的强度。图形工程师指出，这种持续的、高频率的数据传输导致GPU在满负荷和暂停之间快速切换，从而在PC内部产生了电源波动。这些波动通过USB端口传播，干扰了DAC并产生了可听见的噪音。

当通过外部电源为DAC供电后，嗡嗡声随之消失，这一理论得到了证实。解决方案是优化拾取纹理的访问方式：不再每帧复制整个纹理，而是在需要时只下载鼠标光标下的单个像素。这不仅解决了嗡嗡声问题，还提高了GPU性能。
