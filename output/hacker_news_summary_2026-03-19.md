## 4. 他妈的搞个网站

**原文标题**: Have a fucking website

**原文链接**: [https://www.otherstrangeness.com/2026/03/14/have-a-fucking-website/](https://www.otherstrangeness.com/2026/03/14/have-a-fucking-website/)

文章《去他妈的网站》是一篇充满激情、措辞激烈的呼吁，旨在敦促企业和个人创作者建立自己的网站，而非仅仅依赖社交媒体平台。作者认为，尽管社交媒体看似简单且成本低廉，但它是一个“破败的”系统，用户并不拥有自己的内容或粉丝群。

建立网站的主要原因包括：
1.  **更广阔的触及范围：** 并非所有潜在客户都在社交媒体平台上，许多人更喜欢一个简洁、直接的网站来获取如价格和营业时间等信息。
2.  **控制权和稳定性：** 平台可能一夜之间改变规则，让辛苦积累的粉丝变得一文不值，或者在用户毫无追索权的情况下封禁他们。一个网站则提供了一个安全、属于自己的存在，独立于“吸毒成瘾的科技霸主”。
3.  **数据所有权：** 社交媒体平台收集用户数据并通过广告获利；而网站则允许创作者控制自己的信息。
4.  **去中心化：** 文章倡导回归互联网最初的互联网站点概念，摆脱那些不鼓励外部链接的“围墙花园”。

作者还建议建立邮件列表，称电子邮件是“唯一不容易被剥夺的联系方式”。

评论者大多表示赞同，表达了对社交媒体局限性的沮丧以及对“互联网早期时代”的向往。尽管承认创建和维护网站对于非技术用户来说可能是一个“障碍”，但许多人强调，简单、功能齐全的网站就足够了，并且对于真正的在线独立至关重要。

主要政策变化包括广泛的区域规划改革，例如设立垂直混合用途（VMU）区域、在就业和交通枢纽附近进行目标性重新划区，以及放宽附属住宅单元（ADU）的规定。奥斯汀还成为美国最大的取消最低停车位要求的城市。为提高可负担性，该市对包含收入限制单元的项目实施了容积率奖励，并获得了选民批准发行数亿美元的市政债券以资助经济适用房建设。审批流程得到简化，“可负担性解锁”（Affordability Unlocked）等举措进一步放宽了对收入限制项目的开发要求。

奥斯汀继续实施前瞻性改革，包括“HOME”计划，旨在促进双拼屋、三拼屋和附属住宅单元的建设，减少地块最小面积要求，并增加高度灵活性。简化的审批（例如“精简场地规划”Site Plan Lite、AI预审工具）以及允许建造单楼梯中层公寓进一步降低了成本并加速了开发。奥斯汀的经验表明，采取多方面综合策略——即接受公寓建设、侧重可负担性、简化开发流程以及鼓励多样化的住房类型——能够有效缓解住房短缺并提高可负担性。

## 6. 十年滞缓

**原文标题**: A Decade of Slug

**原文链接**: [https://terathon.com/blog/decade-slug.html](https://terathon.com/blog/decade-slug.html)

2016年由Eric Lengyel开发的Slug算法，在今年迎来了它的十周年纪念。这种非常成功的方法直接在GPU上从贝塞尔曲线渲染字体和矢量图形，实现了健壮、快速和高质量的抗锯齿效果。Slug在视频游戏行业（动视、暴雪、Adobe）以及CAD和医疗设备等其他领域获得广泛授权，最初是为C4引擎创建的，现在用于Radical Pie公式编辑器中。

自2017年论文和库发布以来，Slug用于根资格和卷绕数计算的渲染核心保持不变。然而，有几个方面有所发展：“带状分割优化”（band split optimization）被移除，以简化像素着色器并减少一半数据；“自适应超采样”（adaptive supersampling）因对微小文本的益处甚微以及膨胀（dilation）带来的改进而被取消。多色表情符号的渲染也转向独立字形，以获得更好的性能。

主要的进步是2019年新增的“动态膨胀”（dynamic dilation）。它在顶点着色器中自动计算每个顶点的最佳边界多边形扩展，通过在视口空间中将多边形精确地扩展半个像素，确保所有部分覆盖的像素都被光栅化。这消除了手动膨胀设置的需要，防止了小文本的锯齿，并避免了大文本浪费GPU资源。针对此计算提供了数学推导。

关键是，Lengyel宣布Slug专利（2019年获得授权，2038年到期）将从2026年3月17日起献给公共领域。这使得任何人都可以自由地实现该算法，无需担心许可问题。为支持此举，参考顶点和像素着色器（包括动态膨胀）已在GitHub上以MIT许可证发布，这相对于原始论文的代码是一个显著的升级。

## 8. 足够详细的规范就是代码

**原文标题**: A sufficiently detailed spec is code

**原文链接**: [https://haskellforall.com/2026/03/a-sufficiently-detailed-spec-is-code](https://haskellforall.com/2026/03/a-sufficiently-detailed-spec-is-code)

The author argues that "a sufficiently detailed spec is code," challenging agentic coding advocates who claim to generate code purely from specifications. This stance addresses two common misconceptions: first, that specification documents are simpler than the corresponding code; and second, that specification work is inherently more thoughtful than coding.

Using OpenAI's Symphony project as an example, the author demonstrates that its "SPEC.md" is less a specification and more pseudocode or literal code, containing database schemas, concurrency logic, and reference algorithms. This illustrates that for a spec to be precise enough for reliable code generation, it must become highly structured, resembling code itself—a necessary "narrow interface" demanding equivalent precision, as Dijkstra noted.

The author also highlights the unreliability of agentic generation, recounting a failed attempt to generate Symphony in Haskell using Claude Code, which produced buggy and non-functional results. This flakiness, seen even in well-defined specs like YAML, suggests that expanding specifications further only leads to a Borges-esque map as large as the territory it describes.

Regarding thoughtfulness, the author contends that industry pressures to optimize for delivery speed often result in "AI-written slop"—incoherent and superficial documents like sections of Symphony's spec—rather than genuinely critical thought. In conclusion, specifications are not time-saving shortcuts; direct coding is often more efficient if precision is the goal. Unclear input will inevitably lead to unreliable output ("garbage in, garbage out").
