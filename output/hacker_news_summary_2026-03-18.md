**原文标题**: Microsoft's 'unhackable' Xbox One has been hacked by 'Bliss'

**原文链接**: [https://www.tomshardware.com/video-games/console-gaming/microsofts-unhackable-xbox-one-has-been-hacked-by-bliss-the-2013-console-finally-fell-to-voltage-glitching-allowing-the-loading-of-unsigned-code-at-every-level](https://www.tomshardware.com/video-games/console-gaming/microsofts-unhackable-xbox-one-has-been-hacked-by-bliss-the-2013-console-finally-fell-to-voltage-glitching-allowing-the-loading-of-unsigned-code-at-every-level)

2013年的微软Xbox One曾被宣称“无法破解”，但现在终于被一位名叫“Bliss”的安全研究员成功攻破。这标志着首次公开展示的漏洞，该漏洞允许从主机最低级的启动ROM开始，在主机运行的每个层级加载并执行未签名代码。

此次破解利用了一种复杂的基于硬件的技术，称为“电压毛刺”（voltage glitching）。该方法涉及精确操纵主机的电源供应，以诱导CPU执行时产生瞬时错误，从而有效地绕过了保护该主机十余年的强大安全启动链。尽管针对其他Xbox One变体之前也存在漏洞，但Bliss的这项成就首次实现了对原始型号的完全持久控制。

这一突破意味着初代Xbox One现在有可能运行自制应用程序（homebrew）、定制固件、模拟器，甚至Linux等替代操作系统。然而，执行电压毛刺操作是一种高度复杂的硬件修改，需要专用设备和高级技术技能，这使得普通用户难以实现。

尽管这是主机安全史上的一个重要里程碑，也是对游戏保存和自制社区的一大胜利，但考虑到该主机已有十余年历史，其对当前游戏市场的实际影响有限。尽管如此，它代表着一个最具弹性的主机安全系统最终被攻破。

## 8. 困顿十年

**原文标题**: A Decade of Slug

**原文链接**: [https://terathon.com/blog/decade-slug.html](https://terathon.com/blog/decade-slug.html)

《Slug十年》庆祝Slug算法自2016年秋季开发以来的十周年。Eric Lengyel的GPU字体渲染技术首次发表于2017年，已在电子游戏（例如动视、暴雪、Adobe）、科学可视化和CAD等行业广泛授权。Slug直接从贝塞尔曲线渲染文本和矢量图形，优先考虑鲁棒性、速度和高质量的抗锯齿效果，且不依赖纹理贴图。

自2017年发布以来，该算法得到了改进。“带状分割优化”和“自适应超采样”等次要功能已被移除，以简化像素着色器并优化性能。渲染多色表情符号的方法也得到了改进，通过将图层视为独立的字形，效率更高。

最显著的改进是2019年引入的“动态膨胀”（dynamic dilation）。该技术自动在顶点着色器中计算最佳的边界多边形扩展，确保每个部分覆盖的像素都在视口空间中以精确的半像素膨胀进行栅格化。这消除了对易出错的手动膨胀的需求，并防止了GPU资源的浪费。

本文最后发布了一项重大公告：从2026年3月17日起，Lengyel将Slug专利（2019年授予，原有效期至2038年）贡献给公共领域。这使得任何人都可以自由地实现Slug算法，无需许可或担心知识产权问题。参考顶点和像素着色器，包括动态膨胀，现已在GitHub上以MIT许可证发布。

## 9. Rob Pike's Rules of Programming (1989)

**原文标题**: Rob Pike's Rules of Programming (1989)

**原文链接**: [https://www.cs.unc.edu/~stotts/COMP590-059-f24/robsrules.html](https://www.cs.unc.edu/~stotts/COMP590-059-f24/robsrules.html)

Rob Pike's 5 Rules of Programming, established in 1989, provide practical guidelines for efficient software development, primarily focusing on performance, simplicity, and data structures.

Rules 1 and 2 address optimization. Rule 1 advises against guessing performance bottlenecks, as they often occur in surprising places; optimization should only happen once a bottleneck is proven. Rule 2 reinforces this by stating that developers must measure performance before tuning for speed, and only optimize if one part of the code overwhelmingly dominates execution time. These rules echo Tony Hoare's maxim, "Premature optimization is the root of all evil."

Rules 3 and 4 advocate for simplicity. Rule 3 suggests avoiding "fancy algorithms" because they are often slower due to larger constants when the input size ('n') is small, which is usually the case. Rule 4 emphasizes that complex algorithms are more bug-prone and harder to implement. Both rules align with Ken Thompson's "When in doubt, use brute force" and the KISS (Keep It Simple, Stupid) design philosophy.

Rule 5 underscores the critical role of data structures. Pike asserts that if data structures are well-chosen and organized, algorithms will often become self-evident. He concludes that data structures, not algorithms, are central to programming, a principle also noted by Fred Brooks and often summarized as "write stupid code that uses smart objects."

## 18. If you thought code writing speed was your problem you have bigger problems

**原文标题**: If you thought code writing speed was your problem you have bigger problems

**原文链接**: [https://andrewmurphy.io/blog/if-you-thought-the-speed-of-writing-code-was-your-problem-you-have-bigger-problems](https://andrewmurphy.io/blog/if-you-thought-the-speed-of-writing-code-was-your-problem-you-have-bigger-problems)

The article argues that increasing code writing speed, particularly with AI assistants, is a misguided strategy that can worsen software delivery. Citing Eli Goldratt's Theory of Constraints, the author explains that optimizing a non-bottleneck step creates a "traffic jam" of unfinished work, rather than speeding up the entire system.

Faster code output leads to overflowing review queues, slow feedback, context-switching, and rushed approvals. Work-in-progress (WIP) increases, while cycle time (the actual measure of value delivery) deteriorates. This means more code is produced, but less software is shipped. AI-generated code further compounds this by reducing human understanding, making debugging harder.

The real bottlenecks typically lie elsewhere:
1.  **Not knowing what to build:** Rapidly developing the wrong features.
2.  **Post-code completion delays:** Extensive waiting in PR reviews, CI, QA, staging, and deployment pipelines.
3.  **Deploy trust issues:** Fear of deployment leading to riskier, infrequent releases.
4.  **Lack of feedback:** Shipping features without measuring their impact.
5.  **Coordination problems:** Excessive meetings, dependencies, and single points of approval.

To truly improve velocity, organizations should map their value stream, measure cycle time (from commit to user value), eliminate wait states, enforce WIP limits, and listen to engineers. The competitive advantage comes from identifying, building, and delivering the right solutions efficiently, not from writing code faster.
