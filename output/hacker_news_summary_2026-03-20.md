**原文标题**: Austin’s surge of new housing construction drove down rents

**原文链接**: [https://www.pew.org/en/research-and-analysis/articles/2026/03/18/austins-surge-of-new-housing-construction-drove-down-rents](https://www.pew.org/en/research-and-analysis/articles/2026/03/18/austins-surge-of-new-housing-construction-drove-down-rents)

德克萨斯州奥斯汀在2010年代面临住房危机，由于爆炸性增长和供应有限，2010年至2019年间租金飙升近93%。作为回应，奥斯汀从2015年开始启动了全面的政策改革，以刺激住房建设，特别是租赁住房的建设。

主要政策变化包括广泛的区划改革，例如创建垂直混合使用（VMU）类别、在增长区域进行有针对性的重新划分、放宽附属住宅单元（ADU）规定，以及取消全市最低停车要求。该市还通过对包含收入限制单元的开发项目提供容积率奖励来优先考虑可负担性，并批准了用于经济适用房计划的大量市政债券（2018年2.5亿美元，2022年3.5亿美元）。此外，奥斯汀通过“简化场地规划”（Site Plan Lite）等措施简化了开发流程，修订了兼容性标准，并试点使用人工智能工具以加快审批速度。

这些努力取得了显著成效：从2015年到2024年，奥斯汀新增了12万套住房，增长了30%。供应的激增导致租金大幅下降；中位数租金从2021年12月的1,546美元降至2026年1月的1,296美元，尽管人口增长，仍低于全国中位数。老旧的非豪华建筑（C级）的租金下降了11%。

奥斯汀的多方面举措证明了解决监管障碍如何能够成功增加住房供应，改善不同收入群体的负担能力，并扭转租金上涨趋势。该市将继续推行改革以保持这一势头。

## 6. ArXiv 宣布从康奈尔独立

**原文标题**: ArXiv declares independence from Cornell

**原文链接**: [https://www.science.org/content/article/arxiv-pioneering-preprint-server-declares-independence-cornell](https://www.science.org/content/article/arxiv-pioneering-preprint-server-declares-independence-cornell)

ArXiv，这一开创性的预印本服务器，将于2024年1月起从康奈尔大学图书馆的一个中心转变为一个名为ArXiv Inc.的独立法律实体。此项战略举措旨在确保ArXiv的长期可持续性、稳定性和增长能力，使其能够发展成为一个重要的开放科学平台。

尽管ArXiv Inc.仍将是康奈尔大学的全资子公司，保持其机构联系，但它将获得运营自主权。这种独立性对于实现资金来源多元化至关重要，使其能够超越目前由成员机构和赠款组成的模式，转而寻求慈善捐赠和更大规模的赠款，以支持未来的各项举措。

这一转变将促进ArXiv的计划，包括将其用户群和内容扩展到新的科学领域，现代化其技术和基础设施，并开发增强型服务，例如改进的内容审核、与同行评审的链接以及人工智能工具的整合。ArXiv成立于1991年，自2001年起由康奈尔大学托管，目前拥有超过250万份预印本。新的治理结构，包括其独立的董事会，旨在提供所需的灵活性，以便ArXiv有效应对未来发展。

## 7. 足够详细的规范就是代码

**原文标题**: A sufficiently detailed spec is code

**原文链接**: [https://haskellforall.com/2026/03/a-sufficiently-detailed-spec-is-code](https://haskellforall.com/2026/03/a-sufficiently-detailed-spec-is-code)

作者批评了那些声称能从规格说明书生成代码的“智能体编程”倡导者，认为他们的主张具有误导性，并源于两个误解。第一个误解是规格说明书比代码更简单；第二个误解是规格说明书工作本质上更需深思熟虑。

作者以OpenAI的Symphony项目为例，指出其“SPEC.md”并非规格说明书，而是“披着薄薄一层伪装的代码”，其中包含数据库模式、逻辑和显式函数。这表明，要使规格说明书足够精确以可靠地生成代码，它本质上必须成为代码，正如迪杰斯特拉（Dijkstra）就“窄接口”所论证的那样。

作者亲自测试了这一点，尝试使用Claude Code以Haskell语言生成Symphony，结果尽管经过修复，仍然产生了有bug且无法正常运行的结果。这种“不稳定性”凸显出，即使是详细的规格说明书也无法保证生成可用的代码，而为了精确性进一步扩展它们则会变得庞大到难以管理，类似于博尔赫斯的地图。

关于第二个误解，作者认为行业对交付速度的压力常常会损害深思熟虑性，从而导致规格说明书中的“草率”——例如Symphony的`SPEC.md`读起来语无伦次，像是AI生成的。

总之，作者指出规格说明书并非省时工具。如果速度是优先事项，直接编写代码更有效。编程智能体无法从模糊的输入中推断出清晰度或细节；“垃圾进，垃圾出”的原则适用，混乱的思维无法通过智能体得到弥补。

**原文标题**: Noq: n0's new QUIC implementation in Rust

**原文链接**: [https://www.iroh.computer/blog/noq-announcement](https://www.iroh.computer/blog/noq-announcement)

n0 has announced noq (”number 0 QUIC”), a new general-purpose QUIC implementation in Rust with multipath and NAT traversal support, which has powered iroh since v0.96.

noq originated as a hard fork of Quinn, necessitated by iroh's extensive needs for managing multiple paths, NAT traversal, and congestion outside of Quinn's visibility. As iroh pursued deeper structural changes for QUIC multipath and its unique relay-as-a-path architecture, the development trajectories of the two projects diverged significantly. The hard fork allows n0 to implement these fundamental changes without burdening Quinn maintainers, while maintaining a commitment to collaboration where interests align.

Key features of noq include a full, generic implementation of the QUIC Multipath spec, making various network paths (relay, direct IPv4/IPv6) first-class QUIC concepts. This enables intelligent path selection and per-path congestion management, systematically improving latency. It also provides a robust, production-grade implementation of the QUIC NAT traversal draft, battle-tested on iroh devices, which cleans up and enhances reliability by making NAT operations QUIC-aware. noq further integrates QUIC Address Discovery (QAD) to learn public IP addresses, replacing STUN with an encrypted, privacy-enhancing solution. Enhanced QLog support aids debugging, even visualizing multipath packet flows. A new WeakConnectionHandle API facilitates building connection managers.

Already in production with iroh v0.96, noq has also passed interoperability tests with picoquic. n0 plans to evolve noq as a long-term foundation, focusing on further NAT traversal and performance optimizations, and maintaining engagement with the QUIC working group and Quinn team. They invite collaborators working on QUIC, p2p transport, or applications needing robust network performance.
