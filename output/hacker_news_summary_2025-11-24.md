## 1. Fran Sans – 灵感源自旧金山轻轨显示屏的字体

**原文标题**: Fran Sans – font inspired by San Francisco light rail displays

**原文链接**: [https://emilysneddon.com/fran-sans-essay](https://emilysneddon.com/fran-sans-essay)

Fran Sans 是一款由 Emily Sneddon 设计的新型显示字体，其灵感来源于旧金山 Muni Breda 轻轨列车上独特的 LCD 面板目的地显示器。Sneddon 被它们机械化却又富有个性的特点所吸引，这些特点由 3x5 网格上的几何模块构成，体现了旧金山随处可见的实用性与魅力的独特融合。

她的研究包括参观旧金山交通局（SFMTA）电子维修店，技术员 Armando Lumbad 在那里演示了列车操作员如何使用三位数字代码来激活目的地显示。她发现这些显示器是 Trans-Lite, Inc. 公司在 1999 年由工程师 Gary Wallberg 设计的。Wallberg 在设计时注重“足够性”，采用了固定分段，由此创造出虽不完美但充满魅力的字母形式，并省略了 Q 和 X 等不常用字符。

这种工程驱动的简洁性启发了 Sneddon 创造 Fran Sans。她与 Dave Foster 合作，将模块化设计转化为数字字体，开发了三种样式（Solid、Tile 和 Panel）的大写字母、数字和核心标点符号。虽然有些字形并非完美符合 3x5 逻辑，Sneddon 仍保留了原设计的“非正统之处”，例如 N/零中粗斜线和 Z/7 中细斜线，她认为这些特点定义了其独特的字符。

进一步的灵感来源于 Bell Shakespeare 多功能的 Hotspur 字体和 Letterform Archive 的模块化排版范例。

值得注意的是，这些富有启发性的 Breda 显示器将在 2025 年底前逐步淘汰，取而代之的是现代 LED 设备。Sneddon 希望 Fran Sans 能培养人们对赋予我们城市和生活特色的“不完美”的欣赏，并倡导重视魅力而非仅仅追求效率。

## 9. macOS 原生安全隔区支持的 SSH 密钥

**原文标题**: Native Secure Enclave backed SSH keys on macOS

**原文链接**: [https://gist.github.com/arianvp/5f59f1783e3eaf1a2d4cd8e952bb4acf](https://gist.github.com/arianvp/5f59f1783e3eaf1a2d4cd8e952bb4acf)

文章介绍了macOS中一项新发现的原生功能，用于生成和使用由安全隔区（Secure Enclave）支持的SSH密钥，有效取代了第三方工具。这一功能通过`/usr/lib/ssh-keychain.dylib`实现，该文件现在充当一个`SecurityKeyProvider`——这是一种通常用于FIDO2设备的机制。

密钥生成涉及`sc_auth`命令。要生成受生物识别保护的不可导出密钥，可运行命令`sc_auth create-ctk-identity -l ssh -k p-256-ne -t bio`。可以使用`sc_auth list-ctk-identities`列出密钥，使用`sc_auth delete-ctk-identity`删除密钥。

使用SSH时，生成的“私钥”仅是对安全隔区凭证的一个引用。可以使用`ssh-keygen -w /usr/lib/ssh-keychain.dylib -K -N ""`“下载”此引用。为了进行身份验证，`ssh`命令需要指定`SecurityKeyProvider=/usr/lib/ssh-keychain.dylib`。虽然可能会出现PIN码提示，但如果使用生物识别技术，输入空字符串即可。

此外，密钥也可以直接通过`ssh-add -K -S /usr/lib/ssh-keychain.dylib`添加到`ssh-agent`中。为了简化使用，在您的shell配置文件中设置`export SSH_SK_PROVIDER=/usr/lib/ssh-keychain.dylib`，将使该提供程序成为`ssh`、`ssh-add`和`ssh-keygen`的默认提供程序。

文章还描述了一种“可导出”的密钥变体（使用`-k p-256`参数），其中私钥由安全隔区加密以用于备份。虽然不如不可导出密钥安全，但可以通过`sc_auth export-ctk-identity`导出并在其他设备上导入。

## 12. 着色器：如何仅用X和Y坐标绘制高保真图形

**原文标题**: Shaders: How to draw high fidelity graphics with just x and y coordinates

**原文链接**: [https://www.makingsoftware.com/chapters/shaders](https://www.makingsoftware.com/chapters/shaders)

文章《着色器：如何仅用x和y坐标绘制高保真图形》解释道，着色器是在GPU上运行的小巧而强大的程序，是渲染高保真图形的基础。历史上，CPU以逐像素的方式缓慢渲染图形。然而，现代GPU擅长并行处理，能够同时对大量数据进行操作——这非常适合图形处理。

着色器融入图形渲染管线中，主要包含两种类型。**顶点着色器**处理单个顶点，将其3D坐标转换为2D屏幕位置。在GPU的栅格化器确定哪些像素构成几何图形后，**片段（或像素）着色器**接管后续处理。

片段着色器对于计算每个像素的最终颜色至关重要。正是在这里，应用了复杂的视觉效果——如光照、纹理、阴影和反射——将简单的几何图形转化为细节丰富的图像。着色器接受各种输入：逐顶点属性（如位置和颜色）、统一数据（每次绘制调用中的常量，如相机矩阵）以及纹理。

着色器用GLSL（OpenGL着色语言）等语言编写，赋予开发者和艺术家对物体外观的精确控制。它们利用GPU强大的并行处理能力，将基本的几何图形和数学规则转换为现代图形所特有的令人惊叹的实时视觉保真度。

## 30. Intel 386 标准单元逻辑中的异常电路

**原文标题**: Unusual circuits in the Intel 386's standard cell logic

**原文链接**: [https://www.righto.com/2025/11/unusual-386-standard-cell-circuits.html](https://www.righto.com/2025/11/unusual-386-standard-cell-circuits.html)

Intel 386，一款复杂的1985年处理器，采用了标准单元逻辑——一种自动化布局技术——以满足设计期限。这种由软件放置标准化电路（如反相器和门）的方法被证明是成功的。然而，对386的标准单元电路进行研究后，发现了几个意想不到的设计：大型多路复用器、一个非标准放置的晶体管，以及功能不同的“反相器”。

该芯片的寄存器控制逻辑因x86架构的细微差别而复杂，它出人意料地使用了由CMOS开关而非逻辑门构建的大型多路复用器。这些开关，有时辅以反相器进行放大，用于从多个输入中选择一个，以处理三个7位寄存器控制信号的约17种情况。这些多路复用器电路构成了标准单元区域的很大一部分。

一个独特的发现是，在标准单元列之间发现了一个位置异常的PMOS晶体管。作者推测这个“错位”的晶体管是手动修复的一个缺陷，插入它是为了避免在大型机上重新生成整个布局所带来的昂贵且耗时的过程。

最引人入胜的是，一些物理上与反相器相同的单元却并非反相器。最初，由于它们显然会破坏多路复用器输出并短路输入，这令人困惑；但进一步调查显示，这些“假反相器”实际上是独立的NMOS和PMOS晶体管。这允许独立控制，使它们能够将输出拉高、拉低或使其浮空，有效地作为多路复用器的额外情况，而非简单的反相。细微的物理差异，例如未连接的栅极和独立的输入连接，将它们与真正的反相器区分开来。这些发现展示了386开发过程中所采用的实用独创性。

## 43. Rust 的几个核心问题

**原文标题**: Several core problems with Rust

**原文链接**: [https://bykozy.me/blog/rust-is-a-disappointment/](https://bykozy.me/blog/rust-is-a-disappointment/)

作者承认C++存在诸多缺陷，例如未定义行为和隐式复杂性，但认为Rust尽管流行，却存在几个根本性问题。

首先，**Rust的编译速度众所周知的慢**，这是其泛型、单态化和严格的借用检查器等固有设计选择所致，使其比C++更慢。这种设计也导致了Rust的**高复杂性**，迫使开发者即使在处理高级逻辑时，也需应对底层的所有权和借用概念（例如`Arc<Mutex<Box<T>>>`），阻碍了生产力和代码清晰度。

其次，作者质疑Rust**对内存安全的绝对专注**，认为对许多应用而言，偶发性故障总比崩溃要好，并举例嵌入式系统和一次Cloudflare中断事故。他们主张Rust以可靠性和开发者心智负担为代价实现了内存安全，并且无论如何也不是100%安全的。

最后，Rust的内存模型**对于可变共享状态表现不佳**（例如GUI、数据库、操作系统）。使用`Arc<Mutex>`等安全并发原语会因缓存问题而带来性能损失，而`unsafe`块则会破坏内存安全性。这意味着Rust在处理大量可变状态的应用时失去了其核心优势；成功的Rust项目通常依赖于共享只读状态和单向数据流。

总之，作者认为Rust是一种“平庸”的语言。尽管它适用于特定的非交互式、具有不可变数据流的用例，但其固有的缓慢、复杂性以及对可变状态的限制，使其无法成为人们常宣传的通用解决方案，未能在许多实际问题上对C++做出有意义的改进。

## 52. 逆向工程师剖析 macOS 启动链与安全架构

**原文标题**: A Reverse Engineer's Anatomy of the macOS Boot Chain and Security Architecture

**原文链接**: [https://stack.int.mov/a-reverse-engineers-anatomy-of-the-macos-boot-chain-security-architecture/](https://stack.int.mov/a-reverse-engineers-anatomy-of-the-macos-boot-chain-security-architecture/)

本文剖析了Apple Silicon芯片上的macOS启动链和安全架构，强调其根植于硬件的信任机制。该过程始于不可变、掩膜编程的引导ROM（SecureROM），它以最高权限（重置向量）执行，以建立初始系统状态。执行后，该引导ROM变得不可转储。

解密的核心是硬件熔断的组ID (GID) 密钥，这是一个256位AES密钥，特定于芯片类别。引导ROM使用此GID密钥解包包含每个镜像AES密钥的密钥包（KBAG），然后这些密钥用于解密低级引导加载程序（LLB）。GID密钥仅限于硬件，从而阻止了生产固件的离线解密。完整性由公钥加速器（PKA）确保，它利用抗毛刺逻辑验证固件签名，对照内嵌的苹果根CA。

“芯片生产/固件模式”(CPFM) 熔丝决定了芯片的行为。生产设备（CPFM 0x3）强制实行全面安全，禁用调试接口，并使用特定的GID密钥。开发熔断设备（CPFM 0x0/0x1）提供宽松的安全设置、JTAG访问、自定义固件支持以及不同的GID密钥。

除了标准的Armv8-A，苹果还通过指针认证（PAC）等专有功能扩展了ISA。PAC使用五个专用硬件密钥（APIA/B、APDA/B、APGA）对指针（代码、数据、通用）进行加密签名，使得更改指针的内存损坏攻击可被检测，因为CPU会生成一个无效地址。

## 67. Windows ARM64 内部原理：解构指针认证

**原文标题**: Windows ARM64 Internals: Deconstructing Pointer Authentication

**原文链接**: [https://www.preludesecurity.com/blog/windows-arm64-internals-deconstructing-pointer-authentication](https://www.preludesecurity.com/blog/windows-arm64-internals-deconstructing-pointer-authentication)

指针认证码（PAC）是ARMv8.3-A/v9.0-A架构上的一种反漏洞利用功能，它通过将加密签名存储在64位地址的未使用位中来对指针进行签名，并在运行时通过特殊的CPU指令进行验证。Windows ARM64利用PAC用于“指令指针”，主要是在用户模式和内核模式下对返回地址进行签名。

Windows上PAC的启用始于`ntoskrnl.exe`的`KiSystemStartup`。引导加载程序（`winload.efi`）提供初始支持信息和一个由`SymCryptRngAesGenerate`生成的内核指针签名密钥。Windows目前使用ARM的“密钥B”，这是一个128位的值，加载到`APIBKeyLo_EL1`和`APIBKeyHi_EL1`系统寄存器中（EL1表示内核模式权限）。尽管这些寄存器是共享的，但用户模式（EL0）无法直接访问它们。HVCI保护关键的PAC变量，使其只读。

全系统PAC功能由`KiDetectPointerAuthSupport`确定，并在系统初始化期间根据功能标志（`Feature_Pointer_Auth_User__private_featureState`）和注册表覆盖设置完全启用。可以通过`NtQuerySystemInformation`（`SystemPointerAuthInformation`）查询支持和已启用的PAC功能。

对于用户模式，PAC在进程创建（`PspAllocateProcess`）期间是按进程无条件启用的。每个用户模式进程（如果未继承）都会收到自己随机生成的签名密钥，该密钥与全局内核密钥不同。系统寄存器中的活动签名密钥在用户到内核以及内核到用户的转换期间（例如`KiSystemServiceExit`）动态更新。尽管SDK提供了启用/禁用用户模式PAC的选项，但它在Windows上始终处于活动状态，并且没有文档化的方法可以禁用它或审计违规行为。

## 81. Racket v9.0 并行线程

**原文标题**: Parallel Threads in Racket v9.0

**原文链接**: [https://blog.racket-lang.org/2025/11/parallel-threads.html](https://blog.racket-lang.org/2025/11/parallel-threads.html)

Racket v9.0 引入了原生共享内存并行线程，使程序能够充分利用多核硬件，这与先前版本仅支持并发的协程线程相比是一个重大飞跃。创建并行线程只需在 `thread` 调用中添加 `#:pool 'own`，对于并行化任务，这能提供显著的实时加速，其中CPU时间保持相似，但实际运行时间可以减半。

这一发展是在Racket自2017年以来在Chez Scheme（Racket CS）上进行彻底重建之后实现的，这为Racket奠定了坚实的基础，带来了并行垃圾回收器和内存屏障等改进。尽管Racket以前提供了用于粗粒度并行性的 `places` 和用于细粒度任务的 `futures`，但新的并行线程用途更广泛，解决了 `futures` 的局限性，即在参数访问或可变哈希表操作等操作上不会阻塞。

基准测试表明，对于CPU密集型计算和使用非共享数据结构的任务，扩展性表现出色。然而，由于I/O层中的粗粒度锁，I/O密集型操作目前只显示有限的加速。顺序程序可能会由于必要的底层锁机制而经历轻微的性能开销（在特定细粒度情况下高达6-8%）。

Racket现有的同步原语（信号量、通道、`box-cas!`）与并行线程完全兼容。尽管并行线程暴露了机器的弱内存一致性模型，Racket仍通过内部内存屏障保证了内存安全。开发人员预计现有库的兼容性问题将很少见。
